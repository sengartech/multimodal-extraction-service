# Failure Modes

| Failure mode | Detection mechanism | Mitigation | Cost of mitigation |
|---|---|---|---|
| Cross-provider/API failure: primary model returns 429, timeout, or provider outage | API error code, timeout, missing `output_text`, retry counter exceeded | Retry once with backoff. If still failing, route to fallback provider/model or use cached extractor output if available. If no safe fallback exists, return partial result and mark pricing not ready. | Medium. Extra provider integration and occasional duplicate model calls. |
| Model-quality failure: vision misclassifies a fixture or room element | Low confidence, visual observation conflicts with text/drawing, eval checks for unsupported photo-only tasks | Treat photos as existing-state evidence unless another modality supports requested work. Ask clarification instead of creating work tasks from visual evidence alone. | Low to medium. Mostly prompt/eval work, plus possible human review for unclear photos. |
| Drawing interpretation failure: handwritten dimensions are mapped to the wrong wall or unit | Drawing pass returns unitless/low-confidence dimensions, semantic pass flags ambiguous geometry, eval checks unknown dimensions are not used as pricing-grade quantities | Preserve raw dimensions but keep them out of final priced quantities. Add must-have questions for confirmed room dimensions, wall length, and opening sizes. | Low. Conservative output increases follow-up questions but avoids wrong pricing. |
| Schema-evolution failure: downstream pricing engine expects a newer field or changed enum | Schema version mismatch, Pydantic validation error, contract test failure | Version `ScopeBrief`, keep migration notes, add backward-compatible fields where possible, and fail closed if required pricing fields are missing. | Medium. Requires schema migration discipline and compatibility tests. |
| Structured-output failure: model returns valid JSON shape but semantically poor data | Pydantic passes but semantic checks fail: duplicate task IDs, broken `related_task_ids`, vague task names, missing blockers | Add semantic validation between extraction and fusion. Block pricing if references are invalid or critical blockers are missing. | Medium. Additional validation logic and tests. |
| Fusion over-inference: final scope includes work not requested by contractor | Eval checks for photo-only/drawing-only tasks, provenance review, unsupported task categories | Fusion prompt prefers text/audio for requested transformations and uses photos/drawing mainly for context. Unsupported work becomes assumption/question, not task. | Low. Mostly prompt rules and deterministic eval checks. |
| Pricing readiness false negative: system asks too many questions and blocks pricing unnecessarily | Human review of not-ready jobs, high rate of repeated blockers, eval showing missing-but-noncritical questions marked must-have | Tune severity ranking. Separate must-have pricing blockers from nice-to-have details. | Low to medium. Requires calibration data and domain feedback. |
| Pricing readiness false positive: system marks scope ready despite missing critical details | Readiness criteria, must-have question count, eval assertions for wall length, waterproofing, heater spec, room dimensions | Fail closed: any must-have blocker keeps `pricing_ready=false`. Add domain-specific readiness checks for showers, waterproofing, drainage, and electrical work. | Low. Conservative gate may increase manual follow-up. |
| Cost runaway: repeated vision/drawing/fusion calls during debugging or retries | Model-call counters, output file existence, retry logs | Cache typed extractor outputs and make fusion read saved JSON. Use bounded retries and explicit rerun commands. | Low. Requires output management and clear CLI workflow. |
| Regulatory/TVA hallucination: model assigns TVA/category without evidence | Eval assertion that TVA is not claimed, missing source evidence for regulatory fields | Do not infer TVA/regulatory category from weak context. Ask for property/job details or leave field unassigned. | Low. Some downstream manual review remains necessary. |
